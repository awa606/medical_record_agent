# Medical Record Agent 版本演进记录

本文记录 Medical Record Agent 从课程原型到研究级 AI 工程系统的标准版本线。版本线以能力里程碑为准，每个里程碑必须能追溯到 Issue、日志和验证结果。

## 版本管理规则

- 每个功能、修复、研究或重构任务必须先有 GitHub Issue。
- 每天至少维护一个 `/logs/daily/YYYY-MM-DD.md` 工作日志。
- Bug 修复必须在 `/logs/debug/` 下记录问题、根因、修复和验证。
- 每个里程碑必须在 `versions/` 下维护目录和 README。
- 版本目录 `versions/` 只归档关键阶段材料和证据索引，不存放大体积数据、模型权重或隐私数据。
## v0.9.0 病历质量优化

v0.9.0 将当前迭代重点从 ASR 输入质量切换到病历生成质量。此前持续处理 ASR、说话人分离和角色映射，是因为它们会直接影响主诉、现病史、诊断证据和治疗建议的输入可靠性；当前已通过质量门禁避免不可靠角色识别直接污染病历生成，因此本阶段开始加强结构化病历质量。

## v0.9.1 ASR 转写质量修复

v0.9.1 回补 ASR 遗留问题：恢复说话人分离中文规则，合并“嗯、呢、哦”等短口头语和同 speaker 短间隔 utterance，保留医生问句与患者回答的 speaker 边界，并明确单人朗读不能伪造成医生/患者两人。医生端转写区只做轻量清理，将重复的“查看原文”改为“详情”，不再改整体布局。

## v0.9.2 病历质量增强

v0.9.2 继续病历质量主线：在 v0.9.1 稳定 utterance 的基础上优化主诉、现病史、查体、初步诊断、处理建议和证据定位，并把质量摘要用于医生审核辅助，不绕过医生确认。对应 GitHub Issue 为 [#22](https://github.com/awa606/medical_record_agent/issues/22)。

本版本继续增强病历质量评估服务：`SourceSpan` 增加 `segment_id`，字段证据可定位到稳定转写段；`quality_preview.field_quality` 和正式任务 `quality_report.field_quality` 统一输出字段 key、中文名、状态、原因、建议动作和证据数量。占位文本如“待医生查体补充”“未提及，待补充”不再计入完整字段，候选诊断必须保留依据、建议检查、风险提醒和医生确认边界。

前端只补充字段质量标签和“病历质量摘要”，不改变原有审核、导出和 ASR 流程。详细说明见 `docs/病历质量评估说明.md`，每日记录见 `logs/daily/2026-07-13.md`。

## v0.9.4 候选诊断与治疗建议质量优化

v0.9.4 继续后端功能主线，对应 GitHub Issue [#24](https://github.com/awa606/medical_record_agent/issues/24)。本版本不新增 REST API，不改变审核、导出、ASR 和 SSE 流程，只扩展 `quality_preview` 与正式 `quality_report` 中的质量结构。

本版本把候选诊断质量拆成可解释检查项：依据、建议检查、风险提醒、用药边界、补问建议和医生确认边界。每个候选诊断都会输出 `status`、`missing`、`suggested_action` 和 `quality_score` 相关信息，用于说明为什么某个候选诊断还不能进入最终结论。

治疗建议质量同步增强：系统会汇总候选诊断中的建议检查、用药提示、风险提醒和建议补问，并输出 `treatment_safety.status`、`quality_issues` 和 `next_actions`。没有候选诊断时，治疗建议质量标记为 `not_applicable`，避免误拦截仅完成核心字段的病历审核流程。

前端仍遵守 `frontend-freeze`：默认三栏布局不变，只在详情抽屉中补充“建议补问”和治疗建议质量摘要。医生默认界面不增加工程信息，导出前仍必须完成医生审核。

本轮同时补做主链路稳定性复验：`scripts/validate_sse_reconnect.py` 复验 `Last-Event-ID` 断连恢复通过，结果写入 `data/asr_eval/reports/v0_8_11_sse_reconnect/sse_reconnect_report.json`。普通医院 `Windows PC` 复测仍未完成，当前阻塞是缺少真实机器环境与硬件配置采集，因此只在 `logs/debug/2026-07-13_v0_9_4_sse_pc_retest.md` 和每日日志中保留阻塞说明，不把该项写成已完成。

## v0.9.5 最终演示数据与导出体验加固

v0.9.5 对应 GitHub Issue [#25](https://github.com/awa606/medical_record_agent/issues/25)。本版本目标是让评审现场有一条稳定可复现的产品演示路径，并让医生在导出前后都能看到明确反馈。

## v1.0 课程工程交付版

v1.0 对应 GitHub Issue [#26](https://github.com/awa606/medical_record_agent/issues/26)。本版本冻结当前课程工程原型，不新增运行时代码，重点是修复核心交付文档乱码、统一动态端口口径、明确验收边界和演示路径。

本版本将 `docs/scoring/v1_0_final_freeze_checklist.md`、`docs/doctor_workbench_acceptance_v1_0.md` 和 `docs/scoring/v1_0_final_demo_runbook.md` 重写为正常中文，并同步当前端口预检流程。推荐演示端口不再固定写死，而是通过 `python scripts/check_docker_port.py --start 2600 --end 2699 --format env` 获取实际可用端口。

交付边界：系统是课程工程交付版，不声明临床可用；候选诊断、治疗建议和导出结果必须医生确认；医院真实 PC 和边缘端真实部署留作后续实机复测。

## v1.1 医生端信息降噪与产品化收敛

v1.1 对应 GitHub Issue [#30](https://github.com/awa606/medical_record_agent/issues/30)。本版本围绕老师反馈继续产品化收敛：医生不需要理解 ASR 模型、切片、预热、质量评分等工程细节，默认界面只保留关键病历字段、对话转写、候选诊断、治疗方案、诊断证据和确认导出。

本轮不新增后端 API，不修改 ASR、SSE、病历生成、审核和导出流程。医生端默认隐藏顶部 ASR Engine 工程项，右侧 AI 辅助区移除默认质量摘要占位，只保留候选诊断、治疗方案和判断证据三块。质量报告、证据全文、导出阻断原因、转写校正和模型状态继续保留在详情抽屉或显示设置中。

后续路线已整理到 `docs/v1_1_to_v1_4_product_roadmap.md`：`v1.2` 聚焦病历字段抽取增强，`v1.3` 聚焦 ASR 与角色质量继续优化，`v1.4` 聚焦医院 PC 与边缘端实机评估。边缘端部署不作为当前评审阻塞项，需等待医院设备、网络和合规条件明确后再执行。

## v1.2 接口化收敛与字段抽取实验清理

v1.2 对应 GitHub Issue [#31](https://github.com/awa606/medical_record_agent/issues/31)。本版本从“继续堆单页功能”转向“核心能力接口化”，目标是让 ASR、病历字段抽取、草稿构建、质量评估、审核和导出能力能被医生端以外的页面、脚本或其他领域系统复用。

本版本清理了未提交的 `medical_field_extractor/` 实验目录和对应测试，恢复核心 schema、Mock LLM、LLM 记录生成器到稳定链路，避免正式程序继续依赖实验包。字段抽取增强后续应在现有 `app/services/` 与 `app/schemas/` 中收敛实现。

新增接口化能力包括：`GET /api/capabilities` 用于能力发现；`POST /api/records/extract-fields` 用于无副作用字段抽取；`POST /api/records/build-draft` 用于从结构化字段构建草稿、安全校验和质量报告；`POST /api/records/quality` 用于独立质量评估。这些接口不创建 Task、不写入最终病历、不允许直接导出，仍保留医生审核边界。

本版本不修改医生端 UI、不修改 ASR/SSE、不修改正式病历生成、审核和导出接口。领域迁移说明已整理到 `docs/api_capability_contract.md`：可复用 FastAPI 路由、无副作用预览、质量报告模式和 Docker/GitHub 工程流程；迁移到其他领域时需要替换字段 schema、领域知识库、Prompt、质量规则和前端字段标签。

## v1.3.0 追加记录：医生端可用性与病历质量阻塞修复

### 本阶段目标
- 修复真实演示中暴露的医生端阻塞问题：流程条第 5/6 步遮挡、详情抽屉覆盖过宽、右侧 AI 辅助区与操作区挤压、查体字段误抽取。
- 不新增业务 API，不改变 ASR、审核、导出路径；只修复可见问题和字段质量规则。

### 本阶段变更
- 查体字段抽取改为白名单式规则：只接收生命体征、体格检查和客观查体发现；医生追问、患者既往史、家族史、过敏史不再进入查体。
- 处理建议占位文案改为“暂无明确处理建议，需医生结合问诊、查体和检查结果确认”，避免误导为已生成正式医嘱。
- 流程条增加稳定最小宽度和横向滚动，避免小宽度下步骤重叠。
- 详情抽屉默认限制宽度，小屏才全屏，避免遮挡整个工作区。
- 右侧 AI 辅助区改为内部滚动，操作区固定在底部，不再被内容挤压。
- 新增截图验收门禁文档，要求每次前端/流程迭代必须截图、自查并记录结果。

### 验收口径
- 发热/问诊样本中，查体没有真实查体或生命体征时必须保持“待医生查体补充”。
- 1366x768 和 1920x1080 下，流程条、AI 辅助区、详情抽屉不得出现遮挡或重叠。
- 截图证据保存到 `docs/final_report/images/v1_3_0_usability_quality/`。

### 本阶段目标

- 低置信度自动角色默认显示为“说话人 A/B”，不再伪装成确定的医生/患者。
- 详情抽屉保留按整位说话人统一校正入口。
- 全局映射后，同一 speaker 的全部发言同步显示为医生/患者。
## v1.3.6 自动角色分类与录音入口保留

v1.3.6 对应 GitHub Issue `feat: v1.3.6 automatic role classification plan`。本阶段根据用户反馈调整方向：医生端不应要求医生逐句分类角色，程序默认自动完成说话人和医生/患者/其他角色分类，并通过质量门禁兜底。

本轮恢复 `录音生成` 入口，同时将 `Mock 演示` 独立为评审保底入口。录音生成接口保留在医生端入口中，但浏览器麦克风真实录音接入仍放到后续版本；当前主流程仍优先支持上传音频、粘贴文本和 Mock 演示。

新增 `docs/自动角色分类与转写准确率提升计划_v1_3_6.md`，明确提升角色准确率和转写准确率的方法：音频预处理、VAD、医学热词、流式转写加离线校准、CAM++/pyannote/3D-Speaker 说话人分离、医生声纹注册、本地 LLM 整位 speaker 兜底、质量门禁和真实样本标注。当前不优先训练 ASR 大模型，先用评测和工程后处理提高鲁棒性。

## v1.3.4 模型效果提升与鲁棒化路线

v1.3.4 对应 GitHub Issue `docs: v1.3.4 model robustness route`。本阶段不训练 ASR 大模型，而是把模型效果提升路线收敛为可执行工程路径：固定评测集、人工标注、医生声纹注册、全局角色映射、本地 LLM 整位 speaker 角色判断、pyannote/3D-Speaker 候选评测，以及 NeMo Sortformer 高配边缘端研究路线。

文档 `docs/模型效果提升与鲁棒化路线_v1_3_4.md` 明确说明：ASR 负责转文字，说话人分离负责“谁在什么时候说话”，临床角色识别还需要声纹、上下文和人工确认。当前阶段不承诺 100% 自动区分医生/患者，承诺低置信度结果不作为事实，并通过一次全局映射确认保证可用闭环。

## v1.3.3 医生端信息降噪入口收敛

v1.3.3 对应 GitHub Issue `feat: v1.3.3 denoise doctor generation entry`。本阶段回应老师反馈“模型选择医生不一定懂、默认页面文字信息过多”：不再把右上角入口表达为工程化的“输入方式”，而是改为医生更容易理解的“开始生成”。

默认入口展开后只保留三类动作：上传音频、粘贴文本、Mock 演示。Mock 演示作为现场保底流程，不再展示“浏览器录音未接入”的工程解释。模型、切片、质量门禁等细节仍保留在显示设置或详情中，不出现在医生默认主路径。

本轮不修改后端 API、不修改 ASR/SSE、不修改病历生成、审核和导出接口；只更新 `static/doctor.js`、`static/doctor.css` 和静态资源版本号。

## v1.3.2 说话人/角色质量门禁

v1.3.2 对应 GitHub Issue `feat: v1.3.2 speaker role quality gate`。本阶段承认自动模型不能保证 100% 区分医生/患者，产品侧目标改为“低置信度不误导医生、自动结果有指标门禁、失败时退回说话人 A/B/C 和一次全局确认”。

本轮新增 `scripts/validate_speaker_role_quality.py` 和 `tests/test_validate_speaker_role_quality.py`，用于对 ASRResult JSON 执行说话人/角色质量检查。质量门禁覆盖：低置信度临床角色误显示、未完成全局映射、混合语句候选、角色准确率、人工确认率、片段数量和可追踪报告输出。该脚本只用于本地评测和阶段验收，不进入医生端业务流程，不修改 ASR/SSE、病历生成、审核或导出接口。

验收口径：医生/患者不能混在同一稳定 utterance；低置信度不得直接显示为医生/患者；单人朗读样本不得伪造成医患双人；全局映射后同一 speaker 的全部发言必须同步更新。后续 v1.3.3 继续做医生端信息降噪，v1.3.4 继续整理模型效果提升路线。

## v1.4.0 服务端说话人角色质量门禁

本版本在 PR #47 合并、main `verify` CI 通过、release-docs PR 落地后发布。

v1.4.0 将原先只存在于离线脚本中的说话人角色质量检查收敛为服务端统一策略 `SpeakerRoleQualityPolicy`。`ASRResult` 新增 `role_quality` 字段，记录门禁状态、原因和指标；ASR 转写落盘、ASR Session 最终校准、人工角色校正都会写入该质量结果。

正式音频病历生成接口 `/api/audio/{audio_id}/generate-record` 在角色质量未通过时返回 `409`，响应中包含 `role_quality`。外部复用接口 `/api/records/extract-fields` 若传入稳定 `segments`，也必须通过同一门禁；未传 `segments` 的纯文本 demo 路径保持兼容。

本轮统一了低置信度角色、未映射 speaker、混合语句、人工确认率和固定样本角色准确率的判断口径。Ollama `0.88`、两人约束 `0.86` 等低于自动接受阈值的结果不会被服务端当作已通过。混合语句率超过阈值时状态为 `blocked`，不再出现“检测到混合语句但门禁通过”的口径。

验收口径：直接调用 API 不能绕过角色审核；低置信度自动角色不能生成正式病历；混合语句超过阈值不能标记为 `passed`；Mock 演示链路保留但明确标记为确定性 demo 来源。
