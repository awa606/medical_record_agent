# 课程评分规划

本文档用于把当前项目能力映射到课程评分点，便于答辩时说明“为什么这是一个智能体系统”，以及如何体现决策系统和伦理合规设计。

## 智能体设计模式如何拿分

### 可展示内容

- `MedicalRecordOrchestrator` 把任务拆成字段抽取、草稿生成、安全校验、医生审核四个阶段。
- `agent_task_step` 记录每个阶段的输入快照、输出快照、状态和耗时。
- SSE 事件把 Agent 状态实时推送到前端。
- ASR 只负责把音频转换为 `conversation_text`，病历 Agent 主流程保持稳定。
- 医生端 `doctor.html` 展示三栏工作台，调试端 `debug.html` 展示 JSON 和任务日志。

### 得分论证

项目不是简单表单生成文本，而是具备明确的智能体工作流：

1. 感知输入：文本问诊或 ASRResult。
2. 信息抽取：结构化病历字段。
3. 生成行动：病历草稿。
4. 自检约束：安全校验。
5. 人机协作：等待医生审核。
6. 可追踪执行：任务状态、步骤表和审计日志。

### 答辩重点

- 展示 `/api/tasks/{task_id}/steps`，说明每个 Agent 步骤可追溯。
- 展示医生端字段证据，说明不是黑箱输出。
- 强调 AI 只生成草稿，不直接导出最终病历。

## 决策系统设计如何拿分

### 可展示内容

- 字段状态：已确认、待补充、低置信度、候选待确认。
- 安全校验：`passed`、`blocked`、`errors`、`warnings`。
- 候选诊断必须医生确认后才能导出。
- 缺失查体、过敏史、候选诊断未确认等情况会阻止或提醒导出。
- ASR 评测用 CER 和 keyword_recall 辅助选择更可靠的 ASR 引擎。

### 得分论证

项目包含多层决策：

1. 输入来源决策：文本、Mock ASR、FunASR、Qwen3、Online ASR。
2. ASR 质量决策：CER、keyword_recall、missing keywords。
3. 字段可信度决策：缺失字段、低置信度、证据片段。
4. 医疗安全决策：安全校验通过前不允许导出。
5. 人工复核决策：医生确认字段和候选诊断后再导出。

### 答辩重点

- 用 fever 样例展示：关键词可召回，但如果 ASR 角色不可靠，会提示“医生/患者角色需人工校正”。
- 用候选诊断展示：系统给建议，但不会把候选诊断当最终诊断。
- 用导出接口展示：未确认字段或安全校验失败时，导出会被阻止。

## 伦理合规设计如何拿分

### 可展示内容

- 不接真实患者数据。
- 不提交真实 API Key。
- Online ASR 使用环境变量 `ONLINE_ASR_API_URL` 和 `ONLINE_ASR_API_KEY`。
- Qwen3 和 FunASR 依赖独立 requirements，不强制安装。
- 医生端主界面不展示大段 JSON，避免不必要暴露调试信息。
- AI 输出停在医生审核阶段。

### 得分论证

项目体现了医疗 AI 的安全边界：

1. 数据合规：只使用模拟样例和课程音频，不接真实患者隐私。
2. 权限合规：API Key 只从环境变量读取，不写死进代码。
3. 人机边界：AI 生成草稿，医生确认后才可导出。
4. 风险提示：缺失字段、低置信度、角色需校正均显式提醒。
5. 可审计：任务步骤、审计日志和调试 JSON 可追踪生成过程。

### 答辩重点

- 强调“辅助系统”而非“自动诊断系统”。
- 展示安全校验和医生确认按钮。
- 展示文档中“不接真实患者数据、不提交真实 API Key”的约束。

## 建议演示顺序

1. 打开 `/static/index.html`，说明入口分为医生端和调试页。
2. 打开 `/static/doctor.html`，用 fever clean 文本生成病历。
3. 展示左栏字段、字段证据、右栏安全校验和候选诊断。
4. 打开 ASR 评测抽屉，说明 CER 和 keyword_recall。
5. 打开 `/static/debug.html`，展示 ASRResult、Task、Steps、Safety JSON。
6. 尝试确认字段和导出，说明医生审核和安全校验边界。
