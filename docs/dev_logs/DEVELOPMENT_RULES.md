# 开发规范

本文档用于约束后续 Codex、人类开发者和课程汇报材料的同步方式，目标是保证“每次修改都有过程证据，每个评分点都有可展示证据”。

## Issue 驱动

1. 每个新功能、新文档专题、前端重构或评测能力都应先建立 GitHub issue。
2. Issue 标题应写清版本、优先级和目标，例如 `课程评分冲刺 P1：补全 Agent 设计模式与架构图`。
3. Issue 正文至少包含背景、目标、任务、验收标准和不做范围。
4. 修改范围必须和 issue 对齐，避免把无关重构、数据集文件、模型权重或真实隐私数据混入提交。

## 开发日志

1. 每次代码、接口、前端、测试或文档结构发生实质修改，都必须新增或更新 `docs/dev_logs/` 下对应日志。
2. 日志必须使用 `docs/dev_logs/TEMPLATE.md` 的字段，包括修改日期、目标、前置问题、输入、输出、修改文件、关键设计决策、验证步骤、验证结果、未解决问题和下一步计划。
3. 每个 issue 至少对应一篇开发日志；一个 issue 分多次实现时，可以更新同一日志或追加阶段日志。
4. 回顾性日志用于说明版本演进，增量日志用于说明本次具体改动。

## 验证记录

1. 每次提交前必须写清验证步骤，例如 `python -m pytest`、`node --check static/doctor.js`、手动打开 `/static/doctor.html`。
2. 如果某个验证因为缺少可选依赖、网络或模型文件无法运行，日志中必须明确说明原因和风险。
3. 评分材料中的截图、演示路径和代码引用应能回溯到对应 issue 与开发日志。

## 汇报展示

1. 汇报时按 `GitHub Issue -> 开发日志 -> 代码/页面/测试 -> 评分点` 的顺序展示。
2. 智能体设计模式展示 `MedicalRecordOrchestrator`、任务状态、SSE、医生审核和 Agent 架构图。
3. 决策系统展示 Prompt 链、字段缺失判断、ASR 角色校正、安全校验和导出门禁。
4. 伦理合规展示模拟数据、API Key 环境变量、医生最终确认、防 Prompt 注入、审计日志和局限性声明。

## 禁止项

1. 不提交真实患者数据、真实 API Key、医院系统凭据或个人身份信息。
2. 不提交模型权重、下载数据集、大体积音视频运行输出。
3. 不在没有 issue 和日志的情况下合入大范围功能变更。
