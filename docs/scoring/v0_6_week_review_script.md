# v0.6 周评审展示提纲

本文用于 2026-07-10 周评审，重点说明本周从“汇报版”推进到“可评测、可长音频、可前端演示、可本地部署”的工程成果。

## 5 分钟讲法

1. 项目目标

   这是一个面向中文医患问诊的 Medical Record Agent。目标不是自动诊断，而是把音频转写、医生/患者角色校正、病历生成、医生审核和导出串成可追踪流程。

2. 本周重点

   本周主要解决三件事：真实本地 ASR 能否跑，长音频能否稳定，医生端能否连续演示。

3. 技术成果

   我们比较了 FunASR、SenseVoice、Qwen3-ASR。评测指标包括 CER、医学关键词召回、RTF、CPU 和 RSS。当前结论是 SenseVoice/FunASR 更适合普通 Windows PC 演示路线，Qwen3-ASR 保留为研究候选。

4. 长音频处理

   考虑问诊可能达到 16 分钟甚至 30 分钟，系统增加了长音频稳定性测试。SenseVoice 30 分钟整文件不稳定，因此改为 5 分钟切片转写。医生端能看到当前切片、总切片、失败原因和重试提示。

5. 演示路线

   打开 `http://127.0.0.1:2601/static/doctor.html`，按“上传音频 -> 实时转写 -> 角色校正 -> 生成病历 -> 医生审核 -> 导出”展示。现场如果真实模型加载慢，就用 Mock ASR 保底，说明主流程不依赖单个模型。

## 展示顺序

1. 打开 README 或版本记录，说明当前阶段是 `v0.6` 医生端产品化与本地部署。
2. 打开医生端：`http://127.0.0.1:2601/static/doctor.html`。
3. 先走文本导入，展示病历字段、候选诊断、安全提示和医生审核边界。
4. 再走 Mock ASR 音频路线，展示 SSE 分段、角色校正和生成病历。
5. 展示长音频切片截图或现场上传短样本，说明 16/30 分钟压力测试已经完成。
6. 打开评测报告，解释 CER、RTF、RSS 的意义和模型路线选择。
7. 最后说明下一阶段：继续完善真实 ASR 前端验收、界面细节、最终报告和 v1.0 冻结。

## 需要强调的边界

- 当前系统是课程工程原型，不是临床诊断系统。
- 候选诊断、建议检查和用药提示都必须由医生确认。
- Qwen3-ASR 已能跑通，但当前本机 CPU-only 下资源占用高，暂不作为默认演示路线。
- Docker 只做本地和局域网访问，不做公网暴露、HTTPS、登录权限或医院系统接入。
