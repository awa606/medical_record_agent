# v0.7.8 周评审展示方案

本文用于 2026-07-10 周评审。展示重点从 UI 细节转回工程能力：项目已经从“汇报版”推进到可运行、可评测、可长音频、可本地部署、可版本追踪的工程原型。

## 展示主线

- 当前版本：`v0.7.8`
- 医生端地址：`http://127.0.0.1:2601/static/doctor.html`
- 核心一句话：
  “这是一个面向中文医患问诊的 Medical Record Agent，当前已经完成音频/文本输入、实时转写、角色校正、病历生成、医生审核、导出、长音频切片和本地模型评测闭环。”

## 5-8 分钟讲法

1. 项目目标，30 秒

   系统不是自动诊断工具，而是辅助医生完成问诊转写、角色校正、病历草稿生成、候选诊断提示、医生审核和导出。

2. 工程进度，1 分钟

   打开 `docs/版本演进记录.md`，展示从 `v0.1` 到 `v0.7.8` 的版本线。本周重点完成：

   - FunASR、SenseVoice、Qwen3-ASR 本地评测。
   - 中文医患样本评测。
   - 16/30 分钟长音频稳定性测试。
   - 5 分钟切片转写。
   - SSE 切片状态接入医生端。
   - Docker `2601` 本地和局域网访问。
   - GitHub tag 和 Issue 追踪。

3. 医生端主流程，2 分钟

   打开医生端，优先走最稳的文本生成路线：

   - `输入方式 -> 文本生成`
   - 展示病历草稿字段。
   - 展示候选诊断、治疗方案、诊断证据。
   - 强调需要医生确认后才能导出。

4. 实时转写与角色校正，1 分钟

   使用 Mock ASR 或短音频演示：

   - SSE 分段追加到中间转写区。
   - 医生/患者角色可以人工校正。
   - 校正后再进入病历生成流程。

5. 长音频和模型评测，1-2 分钟

   不现场跑 30 分钟音频，展示已有报告和截图：

   - 16/30 分钟长音频稳定性测试已完成。
   - SenseVoice 30 分钟整文件不稳，已用 5 分钟切片规避。
   - FunASR 可作为稳定 fallback。
   - Qwen3-ASR 已跑通，但当前 CPU-only 下资源占用高，保留为研究路线。

6. 部署与版本管理，30 秒

   展示 Docker 和 GitHub：

   - `docker compose up -d --build`
   - `http://127.0.0.1:2601/static/doctor.html`
   - GitHub tag：`v0.7.8`
   - Issue #6 有迭代记录。

## 主动说明的不足

- UI 还不是最终交付版，左侧病历草稿字段高度后续会按内容权重优化。例如现病史、查体、处理建议更高，既往史、过敏史更紧凑。
- 当前知识库是课程模拟知识库，不声明临床诊断正确性。
- 还没有医院电脑实机测试，目前配置建议来自公开资料和本机推断。
- Qwen3-ASR 不作为默认演示模型，默认路线是 `SenseVoice + FunASR fallback + Mock 保底`。
- Docker 目前只做本机和局域网演示，不做公网、HTTPS、登录权限或 HIS/EMR 接入。

## 现场兜底方案

- 真实 ASR 模型加载慢：用 Mock ASR 展示流程，同时展示模型评测报告。
- Docker 访问慢：使用本机浏览器已打开页面演示。
- UI 被指出不够美观：承认当前仍是工程原型，说明下一轮做字段权重布局；本次评审重点是工程功能闭环。
- 医院部署被问到：说明当前先以本机 Docker 和局域网演示为主，功能稳定后再选择网页、桌面软件或边缘端形态。
