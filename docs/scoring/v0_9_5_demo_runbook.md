# v0.9.5 最终演示路径与导出体验验收

## 目标

本轮目标是让评审现场有一条稳定、可复现、可解释的演示路径。默认不依赖真实 ASR 模型必须现场跑完，先展示文本生成和导出闭环，再用 Mock ASR 展示音频链路，真实 ASR 作为可选项。

## 演示地址

- Docker 本机地址：`http://127.0.0.1:2626/static/doctor.html`
- 健康检查：`http://127.0.0.1:2626/health`

## 推荐展示顺序

1. 打开医生端页面，说明系统目标：辅助问诊转写、病历草稿、候选诊断、医生审核和导出，不做自动诊断。
2. 选择 `输入方式 -> 文本生成`，粘贴 `data/demo/v0_9_5_demo_cases.json` 中的 `demo_fever_text`。
3. 点击生成病历，展示左侧病历草稿、右侧候选诊断、治疗方案和诊断证据。
4. 打开详情，展示字段质量、证据来源、建议补问和安全边界。
5. 未确认字段前点击 `确认导出`，展示导出阻断原因。
6. 点击 `确认字段`，再点击 `确认导出`，展示 Markdown 和 Word 导出路径。
7. 如现场时间允许，使用 `Mock ASR` 上传短音频，展示转写、角色校正和生成病历入口。

## 现场兜底

- 真实 ASR 模型加载慢：切换到 `Mock ASR`，说明真实 ASR 有独立评测报告。
- Docker 页面打不开：先执行 `docker compose up -d --build`，再检查 `curl.exe http://127.0.0.1:2626/health`。
- 导出被阻止：这是医生审核边界，点击导出按钮会展示阻断原因；完成确认字段后可导出。
- 普通医院 PC 暂无实机：只展示当前开发机和公开采购资料推断，不伪造成医院实测。

## 验收点

- 文本生成能稳定得到病历草稿。
- 候选诊断和治疗建议必须保留“待医生确认”边界。
- 导出前阻断原因可见，不需要打开调试台。
- 导出成功后可见 Markdown 和 Word 文件路径。
- Mock ASR 可作为现场保底，真实 ASR 不阻塞主流程演示。
