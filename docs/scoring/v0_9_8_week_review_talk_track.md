# v0.9.8 周评审讲稿

## 1. 开场

老师好，我本周继续迭代的是 AI 生成式电子病历辅助系统。当前版本是 `v0.9.7 Release Candidate`，还不是正式 `v1.0`。系统定位是课程工程原型，目标是辅助医生把中文医患问诊音频或文本整理成可审核、可追溯、可导出的电子病历草稿，不替代医生做最终诊断。

## 2. 本周主要工作

本周主要完成三类工作。

第一类是输入质量。系统已经支持文本导入、Mock ASR 保底、FunASR 真实音频转写、SSE 进度和角色校正。对于自动角色识别不稳定的情况，系统不会直接伪装成事实，而是通过质量门禁和医生校正保证后续病历不被污染。

第二类是病历质量。系统现在不仅生成字段，还会判断主诉、现病史、查体、初步诊断和处理建议是否完整、是否有证据、是否需要医生复核。候选诊断和治疗建议也保留医生确认边界，不会输出最终诊断或自动处方。

第三类是工程交付。项目已经有 Docker 本地部署、端口预检脚本、GitHub Issue、tag 和 Release。由于 Windows 可能保留 `2601/2626` 这些端口，所以现在启动前会先扫描 `2600-2699`，选择真实可用端口。

## 3. 演示路线

现场演示我会走最稳定的文本生成路线。

第一步，打开医生端页面。界面分成三栏：左边是病历草稿，中间是对话转写，右边是 AI 辅助和操作区。

第二步，通过输入方式选择文本生成。系统会生成病历草稿，并在字段旁显示质量状态。

第三步，查看右侧候选诊断、治疗方案和诊断证据。这里的所有内容都是候选和建议，必须医生确认。

第四步，点击详情抽屉，展示字段证据、质量原因和导出阻断原因。

第五步，先展示未确认字段时导出被阻止，然后确认字段，再展示导出成功和文件路径。

## 4. 评分表对应

技术打通方面，系统已经打通输入、转写、结构化病历、质量评估、审核和导出。  
系统架构方面，后端使用 FastAPI，核心流程包含 ASR session、Orchestrator、质量评估服务、导出服务和 Docker 部署。  
接口定义方面，ASR、records preview、tasks review/export、export readiness 都已经有明确接口。  
过程记录方面，每天有 daily log，问题有 debug log，版本有 GitHub tag 和 Release。

## 5. 当前不足

当前仍有三个不足。

第一，医院真实 PC 还没有实机，所以性能结论仍需要真实环境复测。  
第二，边缘端还没有真实部署，目前只保留配置建议和后续计划。  
第三，自动角色识别不是 100% 准确，当前通过人工校正和质量门禁来保证输出安全。

## 6. 下一步

下一步是 `v0.9.8` 完成周评审材料和 PPT 提示词，随后进入 `v1.0` 最终冻结。边缘端部署不作为本周阻塞项，会放到 `v1.1` 或后续扩展，在医院设备、网络和合规条件明确后再做。

## 7. 结束语

总体来说，当前项目已经从“汇报版”推进到“可运行、可评测、可追踪、可准备交付”的工程原型。后续重点不再是继续堆功能，而是做最终验收、文档统一和真实环境复测。
