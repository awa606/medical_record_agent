# v1.0 终答辩讲稿

各位老师好，我的项目是 AI 生成式电子病历辅助系统 Medical Record Agent。这个项目的定位不是自动诊断产品，而是一个课程 POC：把文本或音频问诊转换为可审核的病历草稿，同时把候选诊断、安全提醒和证据来源展示给医生，最终确认权始终保留给医生。

系统的核心流程有两条输入路线。第一条是文本问诊，医生粘贴对话后，后端生成结构化字段、病历草稿、候选诊断和安全校验。第二条是音频路线，医生上传 MP3 或 WAV，系统通过 ASR session 和 SSE 把转写段实时推给前端，再进入角色校正和病历生成流程。两个入口最终都复用同一套 records 和 Agent 编排逻辑。

架构上，后端使用 FastAPI，主要接口包括 records、audio、asr_sessions 和 speaker_profiles。ASR 层支持 Mock、FunASR，以及用于评测路线的 SenseVoice、Whisper 和 Qwen3。Agent 层负责字段抽取、草稿生成、安全校验和医生审核。前端是医生端三栏工作台，左侧是病历草稿，中间是对话转写和播放器，右侧是候选诊断、治疗建议、证据和操作区。

本项目后期的主要改进集中在 v0.8.3 到 v0.8.5。首先，上传音频后不再等待整段文件识别完成，而是用 Paraformer streaming 逐步输出文字。其次，加入播放器、时间戳和 Range 支持，医生可以拖动音频，也可以从证据回放到对应片段。第三，离线校准阶段使用 VAD、标点和 CAM++ 对时间戳和说话人进行最终修正。第四，实时预览的病历字段会保留来源 segment id，避免 AI 生成结果无法追溯。

本轮冻结新增了 v0.8.8 diarization 评测证据。这里我只报告两说话人课程样本：`fever_01` 和 `chest_pain_01`。我没有合成三说话人样本，也不会输出三说话人伪成绩。两条样本均补了人工 RTTM，speaker id 固定为 `doctor` 和 `patient`。FunASR CAM++ 的评测结果是：平均 `boundary_f1=0.3708`，平均 `role_consistency=0.9004`；`fever_01` 的 `speaker_count_error=0`，`chest_pain_01` 的 `speaker_count_error=1`。因为本机没有安装 `pyannote.metrics`，DER 和 JER 明确记录为 `not_available`，没有留空，也没有编造。

前端验收方面，我用 Docker 2601 路线复测了医生端。文本生成、Mock ASR、证据面板、FunASR 短音频恢复展示、播放器 Range 和说话人统一角色映射都有截图证据。FunASR 首次运行时出现模型下载和冷启动超过 150 秒的问题，但后端随后完成识别，页面可以恢复 session 并显示 38 条转写段。这个问题归类为真实 ASR 模型运行时耗时，不是前端流程失败。答辩演示时我会先预热 FunASR，现场保底路线是 Mock ASR。

需要特别说明的是，`snakebite_01` 是单人朗读样本，只用于 ASR 和前端流程复测，不纳入 diarization 成绩。系统检测到单真实说话人时，会提示需要人工确认角色，不会伪造成医生和患者两个人。这一点体现了项目边界：模型可以给建议，但不能把不确定事实包装成确定结论。

工程质量方面，项目保留了 pytest、前端 `node --check`、diarization 评测脚本、依赖检查脚本、每日日志、debug 日志、能力证据矩阵和版本演进记录。成果表单由脚本统一生成，避免手工维护多个 Word 表单时出现版本不一致。

最后总结，本项目完成的是一个可运行、可评测、可追踪、可降级的医疗 AI 辅助原型。它的价值不是替代医生，而是把 ASR、生成式 AI 和证据追踪放进一个明确受控的医生审核流程中。后续如果要走向真实场景，还需要补三说话人评测、隐私权限、审计、医院系统接口和临床验证。
