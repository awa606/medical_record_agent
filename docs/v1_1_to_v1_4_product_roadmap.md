# v1.1 到 v1.4 产品化迭代路线

> 历史归档：本文记录 v1.1-v1.4 早期产品化设想。当前唯一路线入口为根目录 [`ROADMAP.md`](../ROADMAP.md)，其中 v1.4.0 已收敛为服务端说话人角色质量门禁。

## 当前阶段

当前版本是 `v1.0 课程工程交付版`，已经具备文本/音频输入、转写、角色处理、病历草稿、质量提示、候选诊断、治疗建议、医生确认、导出、Docker 本地运行和 GitHub Release 追踪能力。

它还不是医院正式上线版本。后续迭代不再盲目堆功能，而是围绕老师反馈做产品收敛、字段抽取增强和真实环境评估。

## v1.1：医生端信息降噪

目标：让医生默认界面更像产品工作台，而不是工程调试面板。

本阶段默认只显示：

- 左侧：病历草稿核心字段。
- 中间：对话转写摘要和简洁进度。
- 右侧：候选诊断、治疗方案、诊断证据。
- 操作区：重新生成、保存草稿、确认字段、确认导出。

默认隐藏：

- ASR 模型细节。
- 切片、耗时、缓存、预热等工程信息。
- 病历质量完整报告。
- 证据全文和导出阻断详细原因。

这些信息不删除，只进入详情抽屉或显示设置，供调试和复核时查看。

## v1.2：病历字段抽取增强

目标：将本地字段抽取实验整理成正式能力，提升主诉、现病史、既往史、过敏史、查体、初步诊断和处理建议的结构化质量。

实施原则：

- 明确提及才填写，不能把未提及写成“无”。
- 主诉优先提炼为“主要症状 + 持续时间”。
- 现病史按起病、发展、伴随症状、处理经过组织。
- 所有字段保留证据定位和质量状态。

## v1.3：ASR 与角色质量继续优化

目标：减少明显错分和碎片化，不追求伪装成 100% 自动正确。

实施原则：

- 继续保留医生全局角色校正。
- 病历生成只消费稳定转写段。
- 角色不可靠时不直接污染病历字段。
- 后续真实多说话人样本继续扩充评测。

## v1.4：医院 PC 与边缘端评估

目标：在拿到真实设备后做实机复测，再决定部署形态。

当前不建议立即部署边缘端，原因是：

- 医院电脑尚未实机采集配置。
- 医院网络、数据合规和本地模型缓存策略尚未确认。
- 当前更关键的是本机 Docker 演示和医生端体验稳定。

本阶段输出：

- 医院 PC 配置采集表。
- 本地模型实机复测记录。
- 边缘端配置建议。
- 是否进入真实边缘端部署的决策说明。

## 每轮完成门禁

每个版本完成时必须具备：

- GitHub Issue。
- 版本记录。
- 能力证据矩阵。
- Daily log。
- 测试结果。
- 如涉及前端，补截图证据。
- 如涉及部署，补 Docker 健康检查。

基础检查命令：

```powershell
pytest -q
node --check static\doctor.js
git diff --check
```

## v1.2 实际执行修订：接口化收敛优先

原计划中 v1.2 主要描述为“病历字段抽取增强”。实际实施时，优先级调整为“接口化收敛 + 字段抽取实验清理”。原因是当前系统已经具备医生端闭环，但核心能力仍主要绑定在医生端完整流程中，不利于后续迁移到其他领域或其他前端。

本轮 v1.2 的实际目标：
- 删除未提交的 `medical_field_extractor/` 实验目录，避免正式程序依赖临时实验包。
- 新增 `GET /api/capabilities`，让外部系统能发现当前可复用能力。
- 新增 `POST /api/records/extract-fields`、`POST /api/records/build-draft`、`POST /api/records/quality`，把字段抽取、草稿构建和质量评估拆成无副作用接口。
- 保持医生端、ASR、审核和导出流程兼容，不改 UI。

后续 v1.3 调整为“接口迁移增强与示例补齐”：补 API 示例请求/响应、跨领域迁移示例、接口契约测试和外部调用说明。更深的医疗字段抽取质量增强继续保留为后续后端质量迭代，不阻塞 v1.2 发布。
