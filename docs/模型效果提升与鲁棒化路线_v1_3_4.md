# v1.3.4 模型效果提升与鲁棒化路线

## 当前结论

现阶段不建议直接训练 ASR 大模型。项目数据量、标注质量、算力和医院实机环境都不足以支撑稳定训练。更现实的路线是先把输入、说话人、角色、质量门禁和人工校正流程做稳，再决定是否训练或微调。

## 为什么不能保证 100% 自动角色区分

医生/患者角色识别包含两个问题：

- 声学问题：谁在什么时候说话，需要 VAD、说话人分离和声纹聚类。
- 语义问题：这个说话人是医生、患者、家属还是其他，需要结合上下文和业务规则。

ASR 只负责把语音转成文字，不等于天然知道“谁是医生”。在单人朗读脚本、多人声音相似、短促“嗯/好/对”、重叠讲话和背景噪声场景中，系统必须保留不确定性，不能把低置信度结果直接显示为事实。

## 短期提升路线

1. 固定评测集
   - `fever_01`
   - `chest_pain_01`
   - `snakebite_01`
   - AliMeeting 三说话人公开样本
   - 后续模拟“医生 + 患者 + 家属”样本

2. 标注与质量门禁
   - 人工标注 RTTM 或 speaker turn。
   - 记录 DER、JER、说话人数量误差、混合语句率、角色准确率、需人工确认率、RTF、RSS。
   - 低置信度时显示“说话人 A/B/C”，不显示确定医生/患者。

3. 医生声纹注册
   - 医生录制 10-30 秒本人语音。
   - VAD 后有效语音不少于 8 秒。
   - 本地只保存声纹 embedding 和元数据，提取后删除原始注册音频。
   - 问诊时优先把匹配声纹的 speaker 锁定为医生。

4. 全局角色映射
   - 转写中先显示说话人 A/B/C。
   - 系统有把握时自动建议 A=医生、B=患者、C=其他。
   - 低置信度只弹一次全局确认，不逐句要求医生修改。

5. 本地 LLM 兜底
   - 使用本机 Ollama `qwen3:4b-instruct` 或同等级本地模型。
   - 输入整位 speaker 的所有稳定发言。
   - 输出结构化 JSON：speaker、role、confidence、reason。
   - 只做整位 speaker 判断，不逐句反复猜测。

## 中期候选路线

| 路线 | 作用 | 适用条件 | 风险 |
| --- | --- | --- | --- |
| FunASR CAM++ | 本地声纹聚类和说话人分离 baseline | CPU 可跑、部署简单 | 会议/医患场景下仍需实测 |
| pyannote.audio | 更成熟的 diarization 方案 | 需要 HuggingFace token 和依赖环境 | 授权、下载、CPU 性能需评估 |
| 3D-Speaker | 中文说话人方向候选 | 适合进一步中文场景对比 | 工程集成成本较高 |
| NeMo Sortformer | 高配/GPU 边缘端研究路线 | NVIDIA GPU、边缘工作站 | 不适合普通医院办公 PC baseline |

## 是否训练模型

只有满足以下条件后才考虑训练或微调：

- 至少 20-50 小时可授权中文医患对话音频。
- 有 speaker turn、角色、病历字段和证据标注。
- 有独立测试集，不用训练数据评估自己。
- 医院或实验室有明确算力和合规条件。
- 当前规则、声纹、LLM 兜底路线仍无法满足评审指标。

当前阶段更应该先补标注、评测和质量门禁，而不是直接训练。

## 本周评审讲法

可以这样回应老师：

> 自动角色区分不能保证 100% 正确，所以我没有把模型结果直接当事实。现在系统会先识别说话人 A/B/C，置信度足够或医生声纹匹配后才显示医生/患者；不确定时只要求医生做一次全局确认。后续提升路线是固定评测集、医生声纹注册、本地 LLM 整位 speaker 判断，再比较 pyannote 和 3D-Speaker，最后才考虑训练。
